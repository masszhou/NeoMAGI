## Findings

### P1 -- Blocker

**F1: `_check_typed_io_validation` exceeds branch block limit (8 > 6)**
- File: `src/growth/adapters/wrapper_tool.py:47`
- `just lint` fails with: `[prod] function_branches src/growth/adapters/wrapper_tool.py:47 _check_typed_io_validation actual=8 limit=6`
- Baseline `.complexity-baseline.json` has `block_findings: []`, so this is a new regression
- Per CLAUDE.md: "src/, scripts/ 出现 single function > 50, nesting > 3, branches > 6 即视为 block"
- **Fix**: Split into two helper functions (e.g. `_validate_schemas()` and `_validate_required_fields()`) or collect validators as a list of `(condition, message)` tuples iterated in a loop

### P2 -- Risk

**F2: In-memory ToolRegistry mutation not guarded by DB transaction in `apply()`/`rollback()`**
- File: `src/growth/adapters/wrapper_tool.py:332-341`
- In `apply()`, `_resolve_and_register(spec, self._registry)` at line 335 modifies the in-memory ToolRegistry before the DB transaction commits. If `update_proposal_status()` at line 336 fails, the DB transaction rolls back but the ToolRegistry retains the registered tool
- Same pattern exists in `_restore_or_disable()` at lines 371-385 for rollback
- The docstring on `apply()` acknowledges this: "the caller is responsible for not leaking the registry state"
- This is consistent with the P2-M1b skill adapter pattern, so it is a known architectural choice, not a new defect
- **Residual risk**: In the failure path, current-state / ledger / registry can drift. Acceptable for V1 given that `apply()` failures are expected to be retried or aborted entirely

### P3 -- Minor

**F3: No type guard on factory return value in `_resolve_and_register()`**
- File: `src/growth/adapters/wrapper_tool.py:227`
- `factory()` result is passed directly to `registry.replace(tool)` without checking `isinstance(tool, BaseTool)`
- If the factory returns a non-BaseTool object, the ToolRegistry will silently accept it, which could cause runtime errors later
- Low probability given that the entire flow is governance-gated (proposal + eval + apply), and the dry-run smoke check validates module importability
- **Suggestion**: Add `if not isinstance(tool, BaseTool): raise TypeError(...)` after line 227

**F4: `_check_dry_run_smoke` only validates module importability, not factory existence**
- File: `src/growth/adapters/wrapper_tool.py:92-108`
- The eval check imports the module but does not verify the factory name exists via `hasattr(mod, factory_name)`. A valid module with a non-existent factory will pass eval but fail at `apply()` time
- Low severity since `apply()` will raise `AttributeError` and the DB transaction will roll back

## Residual Risks

1. **Registry-DB drift on apply/rollback failure** (F2): The in-memory ToolRegistry can diverge from DB state if the DB transaction fails after the registry mutation. This is a known V1 limitation shared with the skill_spec adapter. Mitigation: apply/rollback callers should handle exceptions and restart or reconcile.

2. **Factory validation gap** (F3/F4): The eval dry-run check validates module importability but not factory existence or return type. Malformed `implementation_ref` values that pass eval will fail at `apply()` time with an unhandled `AttributeError`/`TypeError`. The DB transaction will roll back, but the error message may not be user-friendly.

3. **Soft-delete semantics for `remove_active`**: Uses `SET disabled = true` rather than `DELETE`. A re-onboarded wrapper tool with the same `id` will need to handle the `disabled=true` row via the `ON CONFLICT` upsert path. This works correctly with the current `_build_spec_upsert` implementation.
