## F1 Fix Verification

### 1. `SkillStore.transaction()` context manager

**File**: `src/skills/store.py` lines 141-151

```python
@asynccontextmanager
async def transaction(self) -> AsyncIterator[AsyncSession]:
    async with self._db_factory() as session:
        async with session.begin():
            yield session
```

**Verdict**: Correct. `session.begin()` opens a SAVEPOINT-backed transaction. On successful exit, SQLAlchemy auto-commits; on exception, auto-rollbacks. The double `async with` nesting ensures the session is also properly closed after the transaction completes or fails.

### 2. `session` parameter on store methods

All four store methods called from `apply()`/`rollback()` now accept `session: AsyncSession | None = None`:

| Method | Session branch (reuse) | Standalone branch (self-commit) |
|--------|----------------------|-------------------------------|
| `upsert_active()` | Lines 258-260: execute on provided session, no commit | Lines 261-265: own session + commit |
| `update_proposal_status()` | Lines 427-428: execute on provided session, no commit | Lines 429-432: own session + commit |

| `create_proposal()` | Lines 358-362: execute on provided session, return gv | Lines 363-369: own session + commit + return gv |

**Verdict**: Correct. When `session is not None`, methods execute on the caller's session and do NOT call `commit()` -- commit responsibility belongs to the `transaction()` context manager. When `session is None`, backward-compatible standalone commit is used.

Additionally, `update_evidence()` received the same `session` parameter (lines 267-302) even though it is not currently called from `apply()`/`rollback()`. This is forward-looking and consistent.
