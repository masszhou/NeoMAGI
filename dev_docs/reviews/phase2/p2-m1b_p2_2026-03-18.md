## Gate 2 Verification Checklist

### 1. SkillResolver depends only on SkillRegistry protocol

**PASS.** `resolver.py` line 9: `from src.skills.types import SkillEvidence, SkillRegistry, SkillSpec, TaskFrame`. The `__init__` parameter type is `registry: SkillRegistry` (the Protocol), not `SkillStore`. No import of `store.py` or `SkillStore` anywhere in `resolver.py` or `projector.py`.

### 2. Only top 1~3 candidates

**PASS.** `SkillResolver.__init__` accepts `max_candidates: int = 3`. Line 48: `top_k = scored[: self._max_candidates]`. Tests confirm default=3 truncation (`TestTopKTruncation::test_default_max_3`) and custom values (`test_custom_max_candidates`).

### 3. Per-skill max 3 delta, total budget max 9

**PASS.** `SkillProjector.__init__` has `max_delta_per_skill: int = 3`. Module-level `_TOTAL_DELTA_BUDGET = 9`. Projection loop (lines 42-46): per-skill cap via `spec.delta[: self._max_delta_per_skill]`, then total cap via `remaining = _TOTAL_DELTA_BUDGET - len(all_delta)` with `skill_delta[:remaining]` and early `break` when budget exhausted. Tests verify: 3x3=9 fills exactly, 4x3=12 capped at 9, partial inclusion at boundary.

### 4. Resolver/projector do not bypass disabled/preconditions/budget

**PASS.**
- Disabled: `list_active()` in `store.py` filters `WHERE disabled = false`. Resolver starts from `list_active()` output.
- Preconditions: `_preconditions_met()` filters before scoring (line 36). Supports `channel:`, `mode:`, `tool:` prefixes; `not:` and unknown prefixes pass through (open-world assumption, documented).
- Budget: Projector enforces both per-skill and total caps. Tests validate enforcement.

### 5. No embedding, cache, or planner imports

**PASS.** Grep across all `src/skills/*.py` for `embedding`, `cache`, `planner` returns zero matches. No LLM SDK imports in any of the three modules.

### 6. TaskFrame extraction is deterministic (no LLM calls)

**PASS.** `task_frame.py` uses only `re` module. `extract_task_frame()` is a pure function with keyword regex patterns and static thresholds. No async, no network, no LLM calls. Module docstring explicitly states "Pure function, no LLM calls, no network access."

### 7. task_type uses finite StrEnum

**PASS.** `TaskType(StrEnum)` in `types.py` lines 33-41 defines exactly 6 values: `research`, `create`, `edit`, `debug`, `chat`, `unknown`. `TaskFrame.task_type` typed as `TaskType`. Pydantic validation rejects invalid values (test `test_invalid_task_type_rejected`).

### 8. V1 precondition filtering is reasonable

**PASS.** `_preconditions_met()` (lines 115-141) handles four prefixes:
- `channel:<name>`: checks `frame.channel`
- `mode:<name>`: checks `frame.current_mode`
- `tool:<name>`: checks membership in `frame.available_tools`
- `not:<tag>`: pass-through (deferred to eval time)
- Unrecognized: pass-through (open-world, documented)

Case-insensitive comparison via `.lower()`. Tests cover all four prefixes plus the two pass-through cases.

## Additional Checks

### Resolver sorting order

Plan specifies: (1) escalation > (2) evidence quality > (3) delta length.
Implementation `_score()` returns `(escalation_bonus, 0, overlap, delta_penalty)`:
- Position 1: escalation bonus (+1 when `escalation_rules` non-empty AND `risk="high"`) -- **matches plan**
- Position 2: hardcoded `0` placeholder for evidence quality -- **acceptable deviation**: docstring explains evidence is fetched after top-K selection, so pre-sort cannot use it. This is intentional V1 simplification.
- Position 3: tag + capability keyword overlap -- **addition not in plan** but necessary for meaningful matching
- Position 4: `-len(delta)` (shorter preferred) -- **matches plan**

Tests validate: escalation priority with high risk, no bonus with low risk, overlap ranking, shorter delta preference.

### Projector empty input

**PASS.** `project([])` returns `ResolvedSkillView()` with all empty tuples. Test `test_empty_candidates_returns_empty_view` verifies.

### TaskFrame Chinese/English keyword matching

**PASS.** Patterns use `re.compile(..., re.IGNORECASE)`:
- research: `搜索|查找|调研|分析|search|find|research|analyze`
- edit: `修改|重写|更新|edit|modify|update|refactor`
- create: `写|创建|起草|write|create|draft`
- debug: `报错|修复|排查|debug|fix|error|bug`

Rule ordering is correct: edit patterns checked before create (so "重写" matches edit, not create via "写"). 27 parametrized tests cover both languages.

### Risk signal word boundary

The `_HIGH_RISK_PATTERN` uses `\brm\s` for "rm" which correctly avoids "form", "framework", etc. Test `test_rm_without_space_is_not_high_risk` verifies this.

**Finding (low severity):** `drop` and `force` patterns lack word boundaries, causing false positives:
- "dropdown menu" -> high risk (false positive)
- "enforce rules" -> high risk (false positive)
- "reinforce learning" -> high risk (false positive)

Impact: Low. The `risk` field only affects resolver scoring priority (escalation bonus), not blocking or security decisions. V1 over-detection is acceptable (conservative direction). Recommend adding `\b` guards in a future iteration.
