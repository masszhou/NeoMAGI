## Findings

### F1 (P1): `apply()` and `rollback()` lack single-transaction atomicity

**Plan requirement** (Proposed Architecture, section 4):
> `apply()` / `rollback()` 涉及 current-state store 与 ledger 的写入必须在单个 DB transaction 内完成，任一失败全部回滚，不允许 current-state 与 ledger 漂移

**Observed behavior**:

In `src/growth/adapters/skill.py`, `apply()` (lines 296-327) performs:

```python
await self._store.upsert_active(spec, evidence)          # step 1
await self._store.update_proposal_status(version, ...)    # step 2
```

Each `SkillStore` method (`upsert_active`, `update_proposal_status`) opens an independent `async with self._db_factory() as db:` session with its own `await db.commit()`:

- `upsert_active()` at `src/skills/store.py` line 210-235: opens session, executes two statements, commits at line 235
- `update_proposal_status()` at `src/skills/store.py` lines 348-371: opens a separate session, executes, commits at line 371

If step 1 commits but step 2 fails (e.g., DB connection drop, constraint violation), the current-state store (`skill_specs` + `skill_evidence`) will contain the materialized skill, but the governance ledger (`skill_spec_versions`) will still show `status='proposed'`. This is the exact current-state/ledger drift the plan prohibits.

The same issue applies to `rollback()` (lines 329-373), which calls `upsert_active()` or `disable()` followed by `create_proposal()` and `update_proposal_status()` -- all in separate transactions.

**Backend report acknowledgment**: The backend report itself states "store methods each commit independently; full transactional atomicity...deferred". This confirms the deviation is known but not resolved.

**Severity**: P1. The plan explicitly calls this out as a steering constraint (`apply()` / `rollback()` atomicity). Failure here means a partial crash can leave the skill governance state inconsistent, requiring manual reconciliation.

**Recommended fix**: Either (a) have `apply()` and `rollback()` accept/create a single DB session and pass it to the store methods, or (b) add a `SkillStore` method that performs the multi-step write in a single `async with self._db_factory() as db:` block. The adapter should not call multiple independently-committing store methods for a logically atomic operation.

### F2 (P2): `rollback()` marks the *previous applied* version as `rolled_back`, not the *current active*

In `src/growth/adapters/skill.py`, `rollback()` (lines 341-352):

```python
last_applied = await self._store.find_last_applied(skill_id)
if last_applied is not None:
    # Re-materialize previous snapshot
    ...
    await self._store.update_proposal_status(
        last_applied.governance_version,
        GrowthLifecycleStatus.rolled_back,
    )
```

This marks the *previous applied version* (`last_applied.governance_version`) as `rolled_back`, not the *current active version* that is being rolled back from. This may cause confusion in the governance ledger about which version was actually rolled back. However, since `find_last_applied` queries `status='active'` and the last active IS the current active, this is logically correct -- the current active entry gets marked as rolled_back. The naming `last_applied` is slightly misleading but the semantics are correct upon inspection. Downgrading to observation only.

### F3 (P2): `_check_projection_safety()` scans `activation` and `summary` but contract only specifies `delta`

The `_check_projection_safety()` function at `src/growth/adapters/skill.py` lines 105-131 scans `delta`, `activation`, and `summary` for prompt injection patterns. The plan (section 4 bullet on `_check_projection_safety()`) says:

> `delta` 每 skill 最多 3 条...V1 使用静态 blocklist 检测明显 prompt override / prompt injection pattern

Scanning `activation` and `summary` in addition to `delta` is a conservative extension (defense in depth). This does not violate the contract since `SKILL_SPEC_EVAL_CONTRACT_V1` lists `prompt_injection_risk` in `veto_conditions` without restricting which fields are checked. **Not a defect** -- the broader scan is a safety improvement. Logged as observation.

### F4 (P3): No test for partial-failure atomicity

There is no test in `tests/growth/test_skill_adapter.py` that verifies `apply()` or `rollback()` behavior when one store call succeeds and the next fails. Given F1, such a test would currently *fail* to demonstrate correct rollback behavior, but it should be added alongside the F1 fix to serve as a regression guard.

### F5 (P3): `_check_activation_correctness` edge case with empty tags

In `src/growth/adapters/skill.py` line 90, after the `if not tags:` branch adds an error, the code continues to:

```python
tag_set = {t.strip().lower() for t in tags}
```

If `tags` is empty (which is an error), this still creates an empty set and the precondition loop runs harmlessly. No functional bug, but the early-exit pattern would be cleaner.

1. **Clean separation of concerns**: `SkillStore` handles persistence, adapter handles governance logic, engine only orchestrates. No leakage.
2. **Row mappers handle NULL JSONB**: `_row_to_spec` converts `None` JSONB arrays to empty tuples (line 86-93).
3. **Proposal payload validation**: `propose()` validates that `skill_spec` and `skill_evidence` are dicts and parseable as domain objects before creating the proposal (lines 208-223).
4. **Contract pinning is correct**: `evaluate()` pins `SKILL_SPEC_EVAL_CONTRACT_V1` at construction time and includes `contract_id`/`contract_version` in every `GrowthEvalResult`.
5. **Migration is well-structured**: `skill_spec_versions.skill_id` intentionally NOT a FK to `skill_specs.id` (documented reason: proposals may reference not-yet-materialized skills).
6. **Test quality is good**: 46 store tests + 58 adapter tests cover happy paths, error paths, and edge cases. Mock patterns are clean.

### DB schema alignment

- Alembic migration (`a8b9c0d1e2f3_create_skill_tables.py`) matches plan section 2 table definitions exactly
- Column types, defaults, indexes all match
- Downgrade path is correct (dependency-reverse drop order)

## Residual Risks

1. **F1 atomicity gap** (P1): Until fixed, a crash between `upsert_active` commit and `update_proposal_status` commit can corrupt governance state
2. **No integration test with real DB**: All tests use mock sessions. DB-level constraints (FK, unique, type) are not exercised. This is acceptable for unit tests but should be addressed in WP E (per plan)
3. **`find_last_applied` queries `status='active'`**: If multiple ledger entries have `status='active'` for the same skill (which shouldn't happen but is not DB-constrained), the `ORDER BY governance_version DESC LIMIT 1` picks the newest, which is correct but fragile

## Gate Recommendation

**FAIL** -- F1 (P1) must be resolved before gate close. F1 is a violation of an explicit plan steering constraint (section 4: single-transaction atomicity for `apply()` and `rollback()`). The fix is straightforward (consolidate store calls into single-session methods or pass a session through) and should not require rearchitecting.

F2-F5 are P2/P3 and can be tracked as follow-up items.
