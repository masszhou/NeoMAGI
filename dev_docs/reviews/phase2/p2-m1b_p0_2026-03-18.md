- **File**: `src/skills/types.py` L157
- **Impact**: No functional issue. Both produce an empty tuple. The inconsistency is purely stylistic -- Pydantic models use `= ()` and the dataclass could too: `failure_signals: tuple[str, ...] = ()`.
- **Recommendation**: Align to `= ()` for consistency in a future cleanup pass. Not blocking.

**P3-2**: Migration `__init__.py` files are empty (no `__all__` or re-exports).

- **File**: `src/skills/__init__.py`, `tests/skills/__init__.py`
- **Impact**: None. Empty `__init__.py` is the correct minimal pattern for namespace packages.
- **Recommendation**: No action needed. Noted for completeness.

## Residual Risks

1. **JSONB server_default quoting** (`server_default="'[]'"`): This is the standard SQLAlchemy pattern where the Python string provides the SQL literal including its quotes. PostgreSQL will implicitly cast `'[]'` to JSONB for JSONB-typed columns. However, this has not been tested against a real database in this review (no integration test in scope). Risk is low -- this is a well-established SQLAlchemy convention.

2. **No integration test for migration**: The migration itself is not tested (up/down against a real PostgreSQL). This is expected for Phase 0 (types + schema only) and consistent with project practice where integration tests are deferred to later phases that exercise the full stack.
