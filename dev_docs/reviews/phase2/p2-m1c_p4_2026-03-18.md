## 6. Residual Risks

1. **Registry-DB drift on apply/rollback failure** (from G2-F2): The in-memory ToolRegistry can diverge from DB state if the DB transaction fails after the registry mutation. This is a known V1 architectural choice shared with the P2-M1b skill adapter. Mitigation: callers retry or abort the full operation. Acceptable for V1 since these flows are governance-gated.

2. **Entry condition enforcement gap** (from G3-F5): `GrowthCaseSpec.entry_conditions` are declarative strings with no runtime enforcement. The CaseRunner starts any cataloged case without checking conditions. Low risk for P2-M1c (only 2 curated cases). Should be addressed before the catalog grows in P2-M2+.

3. **Factory validation gap** (from G2-F3/F4): The eval dry-run check validates module importability but not factory existence or return type. A well-formed `implementation_ref` pointing to a non-existent factory will pass eval but fail at `apply()` time. The DB transaction rollback prevents data corruption, but error messaging could be improved.

4. **UUIDv7 drift** (from G1-F2): Builder artifact IDs use uuid4 with TODO comments for UUIDv7 upgrade. If UUIDv7 is adopted later, existing IDs mix formats. Low risk since both are valid UUIDs.

None of these residual risks are blocking. All are documented, tracked, and have clear mitigation paths.
