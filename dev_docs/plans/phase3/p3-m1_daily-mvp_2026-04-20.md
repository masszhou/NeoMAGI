# P3-M1 实施计划：Daily MVP

- Date: 2026-04-20
- Status: approved
- Scope: 把 NeoMAGI 从治理实验平台收敛为每天可用的 personal agent，补齐日常对话、provider 选择、web 搜索、memory 实用性与 artifact 管理能力
- Basis:
  - `design_docs/phase3/p3_daily_use_roadmap.md`
  - `design_docs/phase3/p3_daily_use_architecture.md`
  - `decisions/0062-phase3-daily-use-capability-completion.md`
  - `decisions/0063-memory-truth-closure.md`
  - `decisions/0064-artifact-run-metadata-boundary.md`
  - `decisions/0065-run-level-provider-model-selection.md`
  - `CLAUDE.md`

## Goal

交付一个可日常使用的 personal agent MVP：

- 用户可用 OpenAI / Gemini / Claude 进行日常对话，前端可选择 provider 与 model。
- 同一 session 切换 provider / model 后，memory 与 scope 行为不变。
- `web_search` / `web_fetch` 能带来源回答网页问题。
- `memory_search` 在低召回场景有 fallback 扩展，更容易命中真实记忆。
- artifacts / runs 目录约定落地，上传和生成文件有稳定引用位置与 DB 元数据。
- 复杂治理能力（evolution、skill learner、procedure runtime）默认关闭，不干扰日常聊天。
- 产出至少 30 条真实 daily cases。

一句话边界：

**P3-M1 让 NeoMAGI 成为可替代部分 claude.ai / ChatGPT 日常使用的工作入口；不扩展 self-evolution，不新建通用抽象。**

## Current Baseline

- `OpenAICompatModelClient` 覆盖 OpenAI 与 Gemini（OpenAI-compatible endpoint）。不存在 `AnthropicModelClient`。
- `AgentLoopRegistry` 持有 per-provider `AgentLoop`；`ProviderEntry` 记录 `name/agent_loop/model`（1:1 provider→model）。
- `ProviderSettings.active` 只允许 `openai | gemini`。前端无 provider / model 选择 UI。
- `MemorySearcher` 基于 tsvector + `ts_rank`，无 query expansion；无 OR fallback 或 corpus-grounded topic inventory。
- workspace 下无统一 `artifacts/` / `runs/` 目录约定。无 `artifacts` 或 `tool_runs` DB 表。
- memory projection 无自动生成标注，无 rebuild 命令。
- `src/tools/builtins/` 无 `web_search` / `web_fetch` 工具。
- runtime profile 概念不存在；evolution / skill / procedure 组件在 gateway 初始化时按配置条件构建。
- `Settings` 无 `claude` / `anthropic` 子配置。

## Core Decisions

### D1. 先做 daily runtime profile，再添加能力

首先引入 `RuntimeProfile`（`daily` / `growth_lab`），让 gateway 按 profile 决定哪些组件启动。`daily` 默认关闭 evolution、skill learner、procedure runtime。后续 slice 在 `daily` profile 上逐一添加新能力。

理由：profile 隔离保证新能力开发不被冻结组件的兼容性问题拖慢。

### D2. Claude provider 使用 native Anthropic SDK，不走 OpenAI-compatible proxy

新增 `AnthropicModelClient`，直接使用 `anthropic.AsyncAnthropic`。消息 / tool / image / streaming 格式转换在 `AnthropicModelClient` 内完成，不进入 Gateway 或 AgentLoop。

理由：ADR 0065 决定；Anthropic Messages API 与 OpenAI Chat Completions 格式差异大，硬走 proxy 会引入不可控语义损失。

### D3. Model profile 在服务端定义，前端只展示

服务端配置每个 provider 的 model profiles（model id、label、capability flags）。前端通过 API 获取可用 profiles，不允许自由输入 model string。`chat.send` 可携带 `provider` + `model_profile`。

理由：ADR 0065 决定；避免前端绕过白名单指定未配置模型。

### D4. Web search 使用 Brave Search API，web fetch 使用受控 HTTP fetch

`web_search` 调用 Brave Search API 返回候选来源。`web_fetch` 对指定 URL 做受控抓取（timeout、size limit、mime type 过滤、SSRF 防护）。两者从第一天携带 provenance（URL、title、fetched_at、excerpt）。

理由：architecture doc 选择 Brave；provenance 是 P3-M1 硬验收项。

### D5. Memory search query expansion 走 PG-native 路线

低召回时先做 OR fallback，再做 corpus-grounded tsvector lexeme inventory（参数化 `unnest(tsvector_to_array(...))` + `GROUP BY`），不在 hot path 调用 LLM。

理由：architecture doc §4.2 决定；日常搜索不应增加 LLM 调用延迟和成本。

### D6. Artifact metadata 用独立 DB 表管理

新增 `artifacts` 和 `tool_runs` 两张表。文件内容保留在 workspace，不进 DB。不复用 builder artifact 作为总表。

理由：ADR 0064 决定；P3 artifact 范围比已有 builder work memory 更宽。

## Non-Goals

- 不扩展 devcoord / procedure runtime / skill learner。
- 不做通用 bash tool。
- 不做面向多用户的安全模型。
- 不做 PDF / image multimodal 输入（P3-M2）。
- 不做 Python sandbox 执行（P3-M2）。
- 不做 coding task 委派（P3-M2）。
- 不把 LiteLLM / OpenAI-compatible proxy 作为 Claude 主路径。
- 不做会话中途 hot switch provider。
- 不做 embedding 索引（P3-M1 阶段不必选）。
- 不建设通用 artifact platform；按真实需求逐步开启。

## Implementation Slices

### Slice A. Runtime Profile + Daily Mode

**目标**：引入 `RuntimeProfile` 配置，让 `daily` 模式下冻结组件不初始化，且 preflight / readiness / doctor 不被冻结组件耦合。

**涉及文件**：
- `src/config/settings.py`：新增 `RuntimeProfileSettings`，`profile: str = "daily"`（`daily` / `growth_lab`）
- `src/gateway/app.py`：`_build_*` 系列函数按 profile 条件构建组件
- `src/infra/preflight.py`：profile-aware preflight 跳过 soul/growth 检查
- `src/gateway/app.py`：readiness check 按 profile 调整

**要点**：
- `daily`：跳过 `evolution_engine`、`skill_learner`、`skill_resolver`、`skill_projector`、`procedure_runtime`
- `growth_lab`：保持当前全量构建
- `AgentLoop` 构造不变；冻结组件不注入即可
- soul_propose / soul_rollback / soul_status tools 在 `daily` profile 下不注册
- 不引入 runtime 动态切换；profile 是启动时确定的
- **Profile-aware preflight**：`daily` profile 下跳过 `_check_soul_versions_readable`（C9）。`run_preflight()` 接受 profile 参数，或读取 `settings.profile`，按 profile 决定执行哪些检查
- **Soul reconcile 在 daily 下改为非阻塞 best-effort**：当前 `PromptBuilder._layer_workspace()` 每轮从磁盘读 SOUL.md 注入 system prompt（`prompt_builder.py:175-194`），`guardrail.py` 从 SOUL.md 提取 identity anchor 做 pre-LLM 检查。因此 SOUL.md 是 daily runtime 的实际输入，不能完全跳过 reconcile。`daily` profile 下 C11 改为：
  - 仍执行 `reconcile_soul_projection()`（从 DB active version 同步到磁盘文件），但不初始化完整 `EvolutionEngine`；只构造一个轻量级 reconcile-only 路径（直接查 `soul_versions` 表 + 写文件）
  - 如果 `soul_versions` 表不存在或无 active version，reconcile 跳过（使用现有磁盘 SOUL.md），状态为 WARN 而非 FAIL
  - reconcile 失败不阻塞启动（保持当前 `CheckStatus.WARN` 语义）
  - 这样 daily 不初始化 evolution engine（无 propose / apply / rollback），但 SOUL.md 磁盘文件在启动时与 DB 保持一致
- **Profile-aware readiness**：`/health/ready` 在 `daily` profile 下不把 evolution engine 纳入 ready 条件；soul_reconcile 的 WARN 不影响 readiness
- `_REQUIRED_TABLES` 中 `soul_versions` 在 `daily` profile 下降级为可选（表存在不报错，不存在也不阻塞启动）

**测试**：
- gateway wiring 测试：`daily` profile 下 agent loop 不持有 evolution / skill / procedure 依赖
- preflight 测试：`daily` profile 下 C9 跳过（soul_versions_readable 不执行）
- preflight 测试：`daily` profile 下 C11 执行轻量 reconcile — DB 有 active version 时同步到 SOUL.md 文件
- preflight 测试：`daily` profile 下 C11 表缺失或无 active version 时状态为 WARN，不阻塞启动
- preflight 测试：`daily` profile 下 C11 reconcile 失败时状态为 WARN，不影响 readiness
- `growth_lab` profile 下行为与当前一致

### Slice B. Claude Provider (AnthropicModelClient)

**目标**：新增 `AnthropicModelClient`，支持 Claude 基础对话与 streaming。

**涉及文件**：
- `src/config/settings.py`：新增 `ClaudeSettings(BaseSettings)` (`env_prefix="CLAUDE_"`, api_key, model, max_tokens)
- `src/agent/model_client.py`：新增 `AnthropicModelClient(ModelClient)`
- `pyproject.toml`：添加 `anthropic` SDK 依赖

**AnthropicModelClient 职责**：
- 实现 `ModelClient` 的 4 个抽象方法：`chat`、`chat_stream`、`chat_completion`、`chat_stream_with_tools`
- 消息格式转换：OpenAI-style `messages` list → Anthropic Messages API 格式
  - `system` message 提取为独立 `system` 参数
  - `assistant` / `user` message 映射
  - tool_calls → `tool_use` content blocks
  - tool results → `tool_result` content blocks
- Streaming：Anthropic stream events → `ContentDelta` / `ToolCallsComplete`
- Retry + exponential backoff（复用或对齐 `OpenAICompatModelClient` 的 `_with_retry` 策略）
- 返回类型保持与 `OpenAICompatModelClient` 一致（`chat_completion` 返回 `ChatCompletionMessage` 兼容结构，或定义 provider-neutral 中间类型）

**关键映射**：
```
OpenAI tool_calls:
  message.tool_calls[i].id / function.name / function.arguments

Anthropic tool_use:
  content[i].type="tool_use" / content[i].id / content[i].name / content[i].input

OpenAI tool result:
  {"role": "tool", "tool_call_id": ..., "content": ...}

Anthropic tool_result:
  {"role": "user", content: [{"type": "tool_result", "tool_use_id": ..., "content": ...}]}
```

**约束**：
- P3-M1 初期先支持 chat + streaming + tool calling
- Image 支持可作为 Slice B 的 follow-up 或留到 P3-M2
- `chat_completion` 返回值与 `OpenAICompatModelClient` 对齐：如果直接返回 `ChatCompletionMessage` 不可行，需要引入薄 adapter 或 provider-neutral message type（scope 最小化，仅用于 tool_calls 与 content 的统一表达）

**Open question**：
- `chat_completion` 与 `chat_stream_with_tools` 当前返回 OpenAI SDK 类型 (`ChatCompletionMessage`)。`AnthropicModelClient` 无法直接返回该类型。需要在 Slice B 实现时决定：(a) 引入 provider-neutral `ModelResponse` dataclass，两个 client 都返回该类型；(b) 让 `AnthropicModelClient` 构造一个兼容 `ChatCompletionMessage` 的 duck-type 对象。建议 (a)，但实现时再评估影响面。

**测试**：
- 消息格式转换单元测试（OpenAI-style → Anthropic 格式）
- tool_use / tool_result 映射测试
- streaming 事件归一化测试
- mock `anthropic.AsyncAnthropic` 的集成测试

### Slice C. Model Profile Registry + Gateway API

**目标**：支持每个 provider 多个 model profile，并通过 API 暴露给前端。

**涉及文件**：
- `src/config/settings.py`：扩展 `OpenAISettings` / `GeminiSettings` / `ClaudeSettings`，增加 model profiles 配置
- `src/agent/provider_registry.py`：`ProviderEntry` 扩展 model profiles 与 capability flags
- `src/gateway/app.py`：新增 `GET /api/providers` endpoint
- `src/gateway/protocol.py`：`ChatSendParams` 增加 `model_profile: str | None`

**Model Profile 数据模型**：
```python
@dataclass(frozen=True)
class ModelProfile:
    profile: str          # "openai-default", "claude-sonnet"
    model: str            # "gpt-4o", "claude-sonnet-4-20250514"
    label: str            # "GPT-4o", "Claude Sonnet"
    supports_tools: bool
    supports_images: bool
    supports_streaming: bool  # P3-M1 所有内置 profiles = True
    max_context_tokens: int | None  # 用于 per-run compaction 阈值
```

**Capability 范围说明**：
- `supports_streaming`：P3-M1 所有内置 profiles 设为 `True`（当前 message_flow 只走 streaming 路径）。字段必须建模，因为后续可能接入不支持 streaming 的 model（如某些 fine-tune 模型）。如果 `supports_streaming=False` 的 profile 被选中，P3-M1 应 fail-fast 拒绝（不实现 non-streaming fallback 路径），直到有真实需求时再扩展
- `max_context_tokens`：当不为 `None` 时，per-run `BudgetTracker` 使用该值作为 context limit（而非全局 `CompactionSettings.token_budget`）。这确保选择较小 context 的 profile 时，compaction 在正确阈值触发，不会在 provider 侧因超长输入失败。**最小可用预算校验**：profile registry 构造 `ModelProfile` 时，如果 `max_context_tokens` 不为 `None`，必须校验 `max_context_tokens > reserved_output_tokens + safety_margin_tokens`（从 `CompactionSettings` 读取），否则 usable budget 为零或负数，warn/compact 阈值失效。不满足约束的 profile 不注册（startup 日志 WARN），避免运行时异常压缩或不可用

**Profile 配置策略**：
- 每个 provider settings 增加 `profiles: list[dict]` 配置项（或 `models` list）
- `.env` 配置示例：`OPENAI_PROFILES='[{"profile":"default","model":"gpt-4o","label":"GPT-4o",...}]'`
- 或者用 Python 代码内硬编码 per-provider 默认 profiles，`.env` 只覆盖 `api_key` / `model`
- 建议 P3-M1 先用代码定义默认 profiles，后续按需迁移到 `.env` JSON

**Gateway endpoint**：
- `GET /api/providers`：返回 enabled providers、default profile、profiles 列表与 capability flags
- dispatch 路由扩展：`ChatSendParams.model_profile` → 从 `ProviderEntry.profiles` 查找 → 确定实际 model id

**Per-run model 参数传递**：

当前 `AgentLoop._model` 在构造时固定（`src/agent/agent.py:80`），`message_flow` 每轮调用使用该固定值（`src/agent/message_flow.py:258`）。如果不改动，前端选择 model_profile 后实际调用仍会落到 provider 默认模型。

解决方案：**引入结构化 `RunModelSelection`，dispatch 层统一解析并向下游传递**：

新增 frozen dataclass `RunModelSelection`（放在 `src/agent/types.py`，agent / message_flow / compaction 均可直接引用，gateway dispatch 只负责从 RPC params 解析并构造该对象，不引入 agent→gateway 反向依赖）：
```python
@dataclass(frozen=True)
class RunModelSelection:
    provider: str               # "openai" / "gemini" / "claude"
    model_profile: str          # "openai-default" / "claude-sonnet"
    model: str                  # 实际 model id: "gpt-4o" / "claude-sonnet-4-20250514"
    supports_tools: bool
    supports_images: bool
    supports_streaming: bool
    max_context_tokens: int | None  # 用于 per-run compaction 阈值
```

**dispatch 层流程（含不可降级 capability fail-fast）**：

```
dispatch_chat():
  1. _bind_provider(registry, params.provider)
  2. _resolve_model_selection(entry, params.model_profile) → RunModelSelection
  3. ── 不可降级 capability fail-fast（预算预留前） ──
     if not selection.supports_streaming:
         raise GatewayError("CAPABILITY_NOT_SUPPORTED", "streaming required")
     （不消耗预算，不写 reservation/settle 记录）
  4. _reserve_budget_or_raise(selection.model)
  5. handle_message(selection=selection)
  6. _settle_budget(selection.model)
  7. 日志记录 selection.provider / model_profile / model
```

**AgentLoop / message_flow 消费 `RunModelSelection`**：

`RunModelSelection` 必须作为 `RequestState` 的字段存储，贯穿整个 request 生命周期中的所有模型相关路径：

- `AgentLoop.handle_message()` 接受 `selection: RunModelSelection | None` 参数
- `RequestState.selection: RunModelSelection | None` — 初始化时写入，后续所有路径从 state 读取
- **model 路径**：`message_flow` 中所有 `loop._model` 引用改为 `state.selection.model if state.selection else loop._model`，包括 `chat_stream_with_tools()` 调用和日志记录
- **budget / compaction 路径**：
  - `BudgetTracker` 使用 `selection.max_context_tokens`（当不为 `None` 时）作为 context limit override。实现方式：`BudgetTracker.check()` 新增可选 `context_limit_override: int | None` 参数
  - `_apply_compaction()` → `loop._try_compact()` → `compaction_flow._run_compaction()` → `CompactionEngine.compact()`：该路径当前把 `loop._model` 传给 `CompactionEngine`，且 `CompactionEngine` 构造时使用默认模型的 `TokenCounter`。修改：`compact()` 接受可选 `model_override`，`TokenCounter` 使用实际模型的 tokenizer（或者 P3-M1 简化处理：所有 provider 的 token 计数统一使用 `cl100k_base`，与当前行为一致，但记录实际 model 到日志）
- **prompt rebuild 路径**：compaction 后 `_apply_compaction_result()` 重建 system prompt 时，必须从 `state.selection` 读取 capability，确保 `supports_tools=False` 的工具隐藏在 compaction 后仍然生效。`_build_system_prompt()` 的所有调用点使用 `state.selection`，不重新从 loop 构造默认值
- `AgentLoop._model` 保留为 fallback 默认值，不改变构造语义

不做 per-profile AgentLoop（每个 profile 一个 loop 实例），因为同 provider 下多个 profile 共享同一个 `ModelClient` / `ToolRegistry` / `SessionManager`，只有 model / context limit 不同。

**Capability enforcement（ADR 0065 要求）**：

Capability 分两类：**不可降级**（必须在预算预留前 fail-fast）和 **可降级**（进入 handle_message 后以 TextChunk warning 告知用户）。

不可降级（dispatch 层 fail-fast，不消耗预算）：
- `supports_streaming=False`：P3-M1 只有 streaming 路径，不实现 non-streaming fallback

可降级（message_flow 层 gating + 用户告知）：
- `supports_tools=False`：
  - **message_flow 层**：`_resolve_tools_schema()` 在 `selection.supports_tools == False` 时返回空 tools schema，不传 tools 给 `chat_stream_with_tools()`
  - **prompt_builder 层**：当前 `PromptBuilder._layer_tooling()` 独立调用 `ToolRegistry.list_tools(mode)` 把工具列表写入 system prompt。`supports_tools == False` 时，`PromptBuilder.build()` 必须接收 `selection`（或 `effective_capabilities`）并跳过 `_layer_tooling()`，确保 system prompt 中不出现 "Available Tools" 或任何工具名
  - **显式降级通知的协议落点**：当前 AgentEvent 只有 `TextChunk`、`ToolCallInfo`、`ToolDenied`，RPC 协议无 warning/degradation frame。P3-M1 不新增协议 frame，降级通知作为首条 `TextChunk` 输出，前缀为 `[系统提示]`，内容如 `"[系统提示] 当前模型不支持工具调用，本次对话将以纯文本模式进行。"`。无需改动 RPC 协议和前端。后续需要更丰富的 warning 语义时再新增 `RPCWarning` frame
- P3-M1 初期可以简化：所有内置 profiles 都 `supports_tools=True` 且 `supports_streaming=True`；但 gating 逻辑必须实现，因为后续可能配置不支持 tools 的廉价模型

**Provider optionality（Claude-only 启动支持）**：

当前 `OpenAISettings.api_key: str` 是必填字段（无默认值），`PROVIDER_ACTIVE` validator 只允许 `openai | gemini`，`_check_active_provider()` 只识别这两个 provider。如果用户只配置 Claude（`PROVIDER_ACTIVE=claude`，未设 `OPENAI_API_KEY`），Settings 校验或 preflight 会阻塞启动。

改动：
- `OpenAISettings.api_key: str = ""`（改为可选，与 `GeminiSettings` 一致；空字符串 = provider disabled）
- `ClaudeSettings.api_key: str = ""`（新增，同样可选）
- `ProviderSettings.active` validator 扩展允许 `"claude"`
- `_check_active_provider()`（preflight C2）扩展识别 `claude`，检查 `settings.claude.api_key`
- `_build_provider_registry()`（gateway app）只注册 api_key 非空的 providers；active provider 必须在注册列表中，否则 startup fail-fast
- doctor 相关 active provider 检查同步更新

**约束**：
- 未配置 api_key 的 provider 不注册、不出现在 `/api/providers` 返回中
- active provider 必须已配置 api_key（startup fail-fast）
- 每次 run 必须记录 provider / model profile / model（使用实际调用的 model，不是 loop 默认值）
- profile 能力不足时必须显式降级或拒绝，不允许静默失败（ADR 0065）

**测试**：
- `/api/providers` endpoint 返回格式测试
- `ChatSendParams.model_profile` 路由到正确 model 测试
- per-run model override 实际传入 `chat_stream_with_tools` 测试
- 未指定 model_profile 时使用 provider 默认 model 测试
- 未配置 provider 不暴露测试
- **Claude-only 启动测试**：`PROVIDER_ACTIVE=claude` + `CLAUDE_API_KEY=sk-...` + `OPENAI_API_KEY` 未设置 → Settings 校验通过、preflight C2 通过、registry 只含 claude
- **preflight C2 测试**：active provider api_key 为空 → FAIL；active provider 不在注册列表 → FAIL
- **budget 审计测试**：`_reserve_budget_or_raise` 和 settle 日志使用 `actual_model`（来自 profile），不使用 `entry.model` 默认值

**web_search**：

- 参数：`query: str`, `count: int = 5`

**web_fetch**：
- 受控 HTTP GET 抓取页面正文
- 参数：`url: str`, `extract_text: bool = True`
- 限制：timeout（默认 15s）、response size（默认 1MB，流式强制执行）、mime type 白名单（`text/html`, `text/plain`, `application/json`）
- **Response size 流式强制执行**：必须从 httpx 调用层保证真正的 streaming 读取。如果使用 `await client.get(url)`，httpx 会在返回 response 前将 body 完整读入内存，后续 `aiter_bytes()` 只是在内存上迭代，size limit 形同虚设。实现要求：
  - 必须使用 `async with client.stream('GET', url) as response:` 或 `client.send(request, stream=True)` 发起请求，确保 response body 延迟读取
  - Content-Length header 超过 `fetch_max_bytes` 时，直接关闭响应并返回 size limit 错误（快速预检）
  - **所有响应（含有 Content-Length 的）统一走流式累计**：用 `response.aiter_bytes(chunk_size=8192)` 逐块读取，累计字节数超过 `fetch_max_bytes` 时立即 `await response.aclose()` 并返回错误。不能仅靠 Content-Length 预检放行，因为 header 可能错误，且压缩响应解压后可能远超声称大小
  - 禁止在任何路径使用 `response.aread()` 或 `response.text` 一次性读取
- HTML → text 提取（使用轻量库如 `beautifulsoup4` + `html.parser`，或 `trafilatura`）
- 返回：`url`, `title`, `content` (截断), `fetched_at`, `content_length`
- provenance 字段必须在 tool result 中明确区分"原文内容"与"model summary"

**SSRF 防护（多层闭合 + TOCTOU 消除）**：

仅做字符串级别 hostname 黑名单不够；DNS rebinding、IPv6 和 redirect 都能绕过。且如果先解析校验再把原 URL 交给 HTTP 客户端重新解析，DNS rebinding 可在校验和真实连接之间切换解析结果（TOCTOU）。必须实现多层防护并消除双解析窗口：

1. **URL scheme 白名单**：只允许 `http://` 和 `https://`，拒绝 `file://`、`ftp://`、`gopher://` 等
2. **DNS 解析后校验（public global unicast allowlist）**：解析 hostname 得到 A/AAAA 记录后，用 `ipaddress.ip_address()` 校验。采用 **allowlist 策略而非 blocklist**，因为 blocklist 无法覆盖所有非公网地址（例如 `100.64.0.0/10` CGNAT 地址，Python `ipaddress` 中既非 `is_private` 也非 `is_reserved`，会穿过 blocklist）。

   Allowlist 规则：**`ip.is_global and not ip.is_multicast`**（只允许公网全球单播地址），并额外显式拒绝：
   - `is_unspecified`（`0.0.0.0`、`::`）
   - IPv4-mapped IPv6 映射后的内网地址（先 `ipaddress.IPv6Address.ipv4_mapped` 提取内层 IPv4，再对内层 IPv4 执行相同 allowlist）

   已验证（Python 3.13 `ipaddress`）：
   - `100.64.0.1`：`is_global=False` → 被 allowlist 拒绝 ✓
   - `224.0.0.1`：`is_multicast=True` → 被 allowlist 拒绝 ✓
   - `10.0.0.1`：`is_global=False` → 被 allowlist 拒绝 ✓
   - `169.254.169.254`：`is_global=False` → 被 allowlist 拒绝 ✓
   - `8.8.8.8`：`is_global=True, is_multicast=False` → 允许 ✓
3. **Pin resolved address（消除 TOCTOU）**：校验通过后，HTTP 客户端必须使用已校验的 resolved IP 发起连接，不得对原 hostname 二次解析。

   **实现方案（已验证当前依赖 httpx 0.28.1 / httpcore 1.0.9）**：

   继承 `httpcore.AsyncNetworkBackend`，override `connect_tcp(host, port, ...)` 方法。在该方法内用已校验的 pinned IP 替换 `host` 参数，再委托给 `httpcore.AnyIOBackend`（公开 API）发起连接。TLS SNI 和证书校验不受影响，因为 `httpcore.AsyncHTTPConnection` 内部会用原 hostname 调用 `stream.start_tls(server_hostname=original_host)`。

   ```python
   class SsrfSafeBackend(httpcore.AsyncNetworkBackend):
       def __init__(self, resolver: Callable, validator: Callable):
           self._inner = httpcore.AnyIOBackend()  # public API
           self._resolver = resolver    # async hostname → list[IP]
           self._validator = validator  # IP → None (raises if not global unicast)

       async def connect_tcp(self, host, port, *, timeout=None, **kwargs):
           ips = await self._resolver(host)
           for ip in ips:
               self._validator(ip)  # raises if not public global unicast
           pinned_ip = str(ips[0])
           return await self._inner.connect_tcp(pinned_ip, port, timeout=timeout, **kwargs)

       async def connect_unix_socket(self, path, **kwargs):
           raise NotImplementedError("unix sockets not allowed for outbound HTTP")

       async def sleep(self, seconds):
           await self._inner.sleep(seconds)
   ```

   **接入 httpx（已验证 API 签名）**：

   `httpx.AsyncHTTPTransport.__init__` 不接受 `network_backend` 或 `pool` 参数（httpx 0.28.1 确认）。因此不能把 `SsrfSafeBackend` 直接传给 `AsyncHTTPTransport`。

   正确的接入方式是实现完整的 `httpx.AsyncBaseTransport`，内部持有带 `SsrfSafeBackend` 的 `httpcore.AsyncConnectionPool`。实现必须以 httpx 0.28.1 内置 `AsyncHTTPTransport.handle_async_request()` 为模板，保留 extensions 透传（per-request timeout 等）、`AsyncResponseStream` 包装和 `map_httpcore_exceptions()` 异常映射：

   ```python
   from httpx._transports.default import AsyncResponseStream, map_httpcore_exceptions

   class SsrfSafeTransport(httpx.AsyncBaseTransport):
       """httpx transport that pins connections to pre-validated IPs."""

       def __init__(self, resolver, validator, **pool_kwargs):
           backend = SsrfSafeBackend(resolver, validator)
           self._pool = httpcore.AsyncConnectionPool(
               network_backend=backend, **pool_kwargs
           )

       async def handle_async_request(self, request: httpx.Request) -> httpx.Response:
           # Mirror httpx.AsyncHTTPTransport.handle_async_request() exactly,
           # only substituting the pool with our SSRF-safe pool.
           assert isinstance(request.stream, httpx.AsyncByteStream)
           core_request = httpcore.Request(
               method=request.method,
               url=httpcore.URL(
                   scheme=request.url.raw_scheme,
                   host=request.url.raw_host,
                   port=request.url.port,
                   target=request.url.raw_path,
               ),
               headers=request.headers.raw,
               content=request.stream,
               extensions=request.extensions,  # ← per-request timeout etc.
           )
           with map_httpcore_exceptions():  # ← httpcore→httpx exception mapping
               core_response = await self._pool.handle_async_request(core_request)
           assert isinstance(core_response.stream, typing.AsyncIterable)
           return httpx.Response(
               status_code=core_response.status,
               headers=core_response.headers,
               stream=AsyncResponseStream(core_response.stream),  # ← proper stream wrapper
               extensions=core_response.extensions,
           )

       async def aclose(self):
           await self._pool.aclose()
   ```

   注意：`map_httpcore_exceptions` 和 `AsyncResponseStream` 是 httpx 的 private import（`httpx._transports.default`），但在 httpx 0.28.1 中已稳定存在且被 httpx 自身的 `AsyncHTTPTransport` 使用。实现时应固定 httpx 版本范围并在 PoC 测试中覆盖。

   使用方式：`httpx.AsyncClient(transport=SsrfSafeTransport(resolver, validator), follow_redirects=False)`

   - 禁止方案：先解析校验后把原 URL 交给 `httpx.AsyncClient()` 默认 transport 自行解析
   - **依赖声明**：`httpx` 和 `httpcore` 必须同时作为 production 依赖加入 `pyproject.toml`（当前 httpx 仅在 dev 依赖中，httpcore 仅通过 httpx 传递依赖）。`SsrfSafeBackend` 直接 import `httpcore.AsyncNetworkBackend`、`httpcore.AsyncConnectionPool`、`httpcore.AnyIOBackend`（公开 API），必须显式声明 httpcore 版本范围（`httpcore>=1.0.0,<2.0.0`）。`SsrfSafeBackend` 中使用 `httpcore.AnyIOBackend`（公开 API），不使用 `httpcore._backends.anyio.AnyIOBackend`（private module path）

4. **Redirect 重新校验**：`httpx.AsyncClient(follow_redirects=False)`；`web_fetch` tool 手动跟随 redirect 时，对每个 redirect 目标 URL 重新执行 DNS 解析 + IP 校验（`SsrfSafeBackend.connect_tcp` 在每次连接时自动执行校验，所以同一 transport 实例可安全复用），防止 redirect 到内网

### Slice F. Memory Search Query Expansion

**目标**：当 primary tsvector 搜索低召回时，自动做 OR fallback 和 corpus-grounded 扩展。

**涉及文件**：
- `src/memory/searcher.py`：扩展 `search()` 方法
- `src/memory/query_processor.py`：新增 OR query 生成
- 可能新增 `src/memory/topic_inventory.py`：corpus-grounded topic inventory

**扩展策略**（按顺序执行，找到足够结果即停止）：

1. **Primary AND search**：当前行为不变（`plainto_tsquery('simple', :query)`，默认 AND 语义）
2. **OR fallback**：AND 结果 < 阈值时，切换为 OR 语义搜索。

   **SQL 实现约束**：当前 `_build_search_sql` 使用 `plainto_tsquery('simple', :query)`，该函数不识别用户字符串中的 `|` 操作符（所有词默认 AND）。因此 OR fallback **不能**简单地把 query 字符串改成 `word1 | word2 | ...` 再传给同一个 SQL。

   正确方式：OR fallback 使用独立的 SQL query mode，用 `to_tsquery('simple', :or_query)` 替代 `plainto_tsquery`，其中 `:or_query` 由代码安全构造为 `lexeme1 | lexeme2 | ...`（每个 lexeme 经过 `normalize_query()` 分词后提取，用 `|` 连接）。构造过程不使用字符串拼接 SQL，而是先在 Python 侧组装 tsquery 表达式字符串，再作为 bind parameter 传入 `to_tsquery('simple', :or_query)`。

   `_build_search_sql` 需要支持 `query_mode` 参数（`"and"` / `"or"`），根据 mode 选择 `plainto_tsquery` 或 `to_tsquery`。OR mode 的 visibility / scope / source_type 过滤条件与 AND mode 完全一致（复用同一套 WHERE 子句）
3. **Corpus-grounded topic inventory**：仍低召回且 query 是宽泛词时，从用户可见 memory corpus 中提取高频主题词，用这些真实词做二次搜索
4. **Retrieval tags**：利用 `memory_entries.tags` 存放 indexer 提取的 top-N 关键词，tags 匹配作为补充召回

**Topic inventory 可见性边界（P1 安全约束）**：

`ts_stat` 直接对全表 `search_vector` 做统计会跨 scope / principal / visibility 泄露记忆主题词。Topic inventory 的子查询必须复用 `MemorySearcher._build_search_sql` 中相同的 scope_key + principal_id + visibility 过滤条件。

**禁止使用 `ts_stat`**：`ts_stat()` 接收文本 SQL，SQLAlchemy bind parameters 不会进入 dollar-quoted 字符串；实现者要么得到不可执行 SQL，要么转向字符串插值 scope/principal，引入 SQL 注入风险。

**必须使用参数化 unnest + GROUP BY 实现**：
```sql
SELECT word, count(*) AS ndoc
FROM (
    SELECT unnest(tsvector_to_array(search_vector)) AS word
    FROM {schema}.memory_entries
    WHERE scope_key = :scope_key
      AND (
          (COALESCE(visibility, 'private_to_principal') = 'private_to_principal'
           AND (principal_id = :ctx_principal_id OR principal_id IS NULL))
          OR
          (visibility = 'shareable_summary'
           AND principal_id = :ctx_principal_id)
      )
) t
GROUP BY word
ORDER BY ndoc DESC
LIMIT :topic_limit
```

该 SQL 使用标准 SQLAlchemy `text()` bind parameters（`:scope_key`、`:ctx_principal_id`、`:topic_limit`），与 `_build_search_sql` 的可见性过滤条件完全一致，不存在注入风险。`tsvector_to_array` 是 PostgreSQL 内置函数（PG 12+）。

关键不变量：**topic inventory SQL 的 WHERE 子句必须与 primary search SQL 的可见性过滤保持结构一致**，由 code review 和测试双重保证。建议将可见性 WHERE 片段提取为 `_visibility_filter_sql()` 共享方法，避免两处手动同步。

**返回结果扩展**：
- 增加 `query_mode` 字段（`"primary"` / `"or_fallback"` / `"topic_expansion"`），方便调试
- 增加 `query_variants` 字段，记录实际使用的查询变体

**约束**：
- 扩展词必须来自用户自己的 memory corpus，且只来自当前 scope / principal 可见的 entries
- 不在 hot path 调用 LLM
- topic inventory 查询结果可做短期缓存（同一请求内）
- 不新增独立 `memory_topics` tool

**测试**：
- AND 有结果时不触发 fallback
- AND 无结果时 OR fallback 能命中（**验证 SQL 真实 OR 语义**：插入两条 entries 分别只包含 query 中的一个词，AND search 返回空，OR search 命中两条）
- OR fallback 使用 `to_tsquery` 而非 `plainto_tsquery`（断言生成的 SQL 包含 `to_tsquery`）
- 宽泛查询走 topic inventory 路径
- query_mode / query_variants 返回正确
- **topic inventory 可见性隔离测试**：不同 scope_key / principal_id 的 entries 的主题词不泄露到对方的 topic inventory 结果中
- **topic inventory SQL 注入测试**：恶意 scope_key（如含 SQL 元字符 `'; DROP TABLE --`）不影响查询结构

**Artifact integrity check（对齐 ADR 0064）**：
- `ArtifactStore` 提供 `verify_integrity(artifact_id, *, deep: bool = False)` 方法：
  - 默认检查：DB 记录存在、`status` 可用、path 指向的文件存在
  - `deep=True`：重新计算文件 `sha256` 并与 DB baseline 比对
- **Doctor 与 verify 的写入边界**：当前 `src/infra/doctor.py` 约定 "All checks are read-only — doctor never writes to DB or files"。Artifact integrity 需要写入 `integrity_status` 和 `last_verified_at`，因此：
  - `doctor` 中的 artifact check 保持 read-only：只报告 drift（missing / modified / unreadable），不写 DB
  - `integrity_status` / `last_verified_at` 的更新由独立 CLI 命令 `artifacts verify` 执行，该命令调用 `ArtifactStore.verify_integrity()` 并写入 DB
  - doctor 的 artifact check 调用 `ArtifactStore` 的只读检查方法（返回检查结果但不更新 DB），与 `verify` 命令调用同一核心逻辑但分离写入
  - 更新 `doctor.py` 文件头注释和 docstring，明确 "artifact integrity check 由独立 verify 命令负责写入状态"
- `doctor --deep`：对 retained artifacts 做全量 `sha256` 校验并报告 drift，但仍不写 DB
- 显式修复命令预留（`artifacts verify [--deep]` / `artifacts reconcile --mark-missing` / `artifacts accept-current-hash`），P3-M1 至少实现 `artifacts verify --deep`

**Artifact path workspace confinement**：

`artifacts.path` 是 workspace-relative canonical path，integrity check / doctor --deep 会读取并 hash 这些路径指向的文件。如果 path 可被写入绝对路径、`..` traversal 或 symlink escape，就可能读取 workspace 外文件。

`ArtifactStore` 和 `src/artifacts/workspace.py` 必须在写入和读取时集中做 path confinement：
- 写入时：`path` 必须是相对路径（拒绝以 `/` 开头的绝对路径）；resolve `workspace_dir / path` 后，确认 resolved path 以 `workspace_dir.resolve()` 为前缀（拒绝 `..` traversal）；resolved path 必须落在允许的 bucket 目录下（`artifacts/uploads/`、`artifacts/web/` 等）；如果 resolved path 是 symlink，resolve 后重新检查前缀约束
- 读取 / integrity check 时：同样做 resolve + 前缀检查后再读文件
- 路径校验逻辑提取为 `_validate_artifact_path(workspace_dir, path) -> Path`，store 和 workspace 模块共用

**约束**：
- `artifacts.source_run_id` 不加硬外键
- `stdout_tail` / `stderr_tail` 入库前截断（默认最后 16 KiB）
- tail 入库前做基本 secret redaction
- P3-M1 不实现 artifact 清理策略
- 不复用 builder artifact 表
- `tool_runs.log_path` 同样做 workspace confinement：`ToolRunStore` 写入 `log_path` 时必须校验 workspace-relative、resolve 后前缀在 `workspace/runs/` 或 `workspace/artifacts/logs/` 下，拒绝绝对路径 / `..` traversal / symlink escape。复用 `_validate_artifact_path()` 或提取为通用 `_validate_workspace_path(workspace_dir, path, allowed_prefixes)` 共享

**测试**：
- store CRUD 测试
- unique constraint on `path` 测试
- artifact write + sha256 计算测试
- workspace 目录创建幂等性测试（包含所有 bucket 子目录）
- integrity check：文件存在 → `ok`
- integrity check：文件缺失 → `missing`
- integrity deep check：文件内容变化 → `modified`
- doctor artifact check 集成测试
- **path confinement 测试**：
  - 绝对路径（`/etc/passwd`）→ 拒绝
  - `..` traversal（`../../../etc/passwd`）→ 拒绝
  - symlink escape（symlink 指向 workspace 外）→ 拒绝
  - 不在允许 bucket 下的路径 → 拒绝
  - 合法 workspace-relative path → 允许
- **tool_runs.log_path confinement 测试**：
  - 绝对路径 → 拒绝
  - `..` traversal → 拒绝
  - 不在 `workspace/runs/` 或 `workspace/artifacts/logs/` 下 → 拒绝
  - 合法 log_path → 允许

### Slice I. Memory-Artifact Link

**目标**：`memory_append` 可接收 artifact_ids，写入 ledger metadata，校验通过 `ArtifactStore` 执行；ledger 读取路径补齐 metadata 返回。

**涉及文件**：
- `src/memory/writer.py`：`MemoryWriter.append_daily_note()` 新增 `metadata: dict | None` 参数，透传至 `ledger.append()`；`MemoryWriter` 构造增加可选 `artifact_store: ArtifactStore | None` 依赖注入
- `src/memory/ledger.py`：
  - `get_current_view()` SELECT 列表增加 `metadata`，返回 dict 包含该字段（当前未 SELECT metadata，reindex 会丢失）
  - `get_entries_for_parity()` SELECT 列表增加 `metadata`，返回 dict 包含该字段（当前未 SELECT metadata，parity check 无法看到 artifact_ids）
- `src/tools/builtins/memory_append.py`：tool schema 增加 `artifact_ids` 可选参数；tool execute 构造 `metadata={"artifact_ids": [...], "relation": "supporting_source"}` 传给 writer
- `src/gateway/app.py`：`MemoryWriter` 构造时注入 `ArtifactStore` 实例

**写入链路改动**：

当前写入链路 `memory_append tool → MemoryWriter.append_daily_note() → MemoryLedgerWriter.append()` 存在以下 gap：
- `append_daily_note()` 不接受也不传 `metadata` 参数（`writer.py:158-162`）
- `ledger.append()` 接受 `metadata` 但上游从未传入

改动：
1. `memory_append` tool 构造 metadata dict（含 `artifact_ids`），传给 `append_daily_note(metadata=...)`
2. `append_daily_note()` 新增 `metadata` 参数，透传至 `self._ledger.append(metadata=metadata)`
3. 写入前通过注入的 `ArtifactStore.get(artifact_id)` 校验每个 artifact_id 存在
4. 校验 artifact 的 `principal_id` / `visibility` 允许当前 memory 引用

**读取链路改动**：

当前读取链路 `get_current_view()` 和 `get_entries_for_parity()` 不 SELECT `metadata` 列，导致：
- `reindex_from_ledger()` 重建 `memory_entries` 时丢失 metadata
- parity check 比对时看不到 metadata，无法验证 artifact_ids 一致性

改动：
1. `get_current_view()` 的 SQL SELECT 列表增加 `metadata`，返回值中包含该字段
2. `get_entries_for_parity()` 的 SQL SELECT 列表增加 `metadata`，返回值中包含该字段
3. Slice H 的 `rebuild-projection` 从 `get_current_view()` 获取 metadata，渲染时保留 artifact 引用信息

**依赖边界**：
- `MemoryWriter` 如果构造时未注入 `artifact_store`（向后兼容 / 测试场景），则 `artifact_ids` 参数必须为 `None`，传入非空列表时 fail-fast
- ledger metadata 格式：`{"artifact_ids": [...], "relation": "supporting_source"}`
- 不新增 `memory_artifact_links` 表

**测试**：
- 带 artifact_id 的 memory 写入成功（需要 `ArtifactStore` fixture）
- 不存在的 artifact_id 拒绝写入
- principal / visibility 不匹配时拒绝
- 未注入 `artifact_store` 时传入 `artifact_ids` fail-fast
- metadata 格式正确
- `get_current_view()` 返回 metadata 字段
- `get_entries_for_parity()` 返回 metadata 字段
- projection rebuild 后 artifact_ids metadata 保留

## Risks

### R1. Anthropic SDK 消息格式差异导致 tool calling 兼容性问题

风险：Anthropic Messages API 的 tool_use / tool_result 格式与 OpenAI 差异大，转换层可能遗漏边界情况。

缓解：Slice B 实现后立即对 `tool_concurrency` / `tool_runner` 的现有测试做 Claude provider 回归；重点覆盖 multi-tool-call、空 content + tool_calls、parallel tool calls。

### R2. ModelClient 返回类型不一致

风险：`chat_completion` 当前返回 OpenAI SDK 的 `ChatCompletionMessage`，`AnthropicModelClient` 无法直接返回该类型，可能需要引入 provider-neutral 中间类型。

缓解：Slice B 评估影响面后决定方案；如果影响面大，provider-neutral type 优先最小化（只包含 `content` 和 `tool_calls`），不过度设计。

### R3. Brave Search API 限额或质量不满足 daily use

风险：免费层 rate limit 或结果质量不足。

缓解：P3-M1 先验证；如果不满足，fallback 到 DuckDuckGo instant answers 或其他免费 API。`web_search` tool 内部抽象搜索后端，方便替换。

### R4. Memory query expansion 误扩大召回或泄露跨 scope 主题

风险：OR fallback 或 topic inventory 返回大量不相关结果；topic inventory 子查询未带 scope/principal/visibility 过滤导致跨用户主题词泄露。

缓解：扩展只在 primary search 低召回时触发；结果标记 `query_mode`，方便调试和回调。topic inventory 限制候选词数量。topic inventory SQL 必须复用 `_build_search_sql` 中相同的可见性条件，由 code review 和专项隔离测试保证。

### R5. 前端改动引入 regression

风险：provider selector 改动影响现有聊天流程。

缓解：现有前端测试回归 + 手动验证 golden path（发送消息、streaming、tool call 折叠）。

### R6. Per-run model override 影响面大于预期

风险：`handle_message` / `message_flow` 中 `loop._model` 的引用分散在多处（streaming、compaction、budget tracking），per-run override 可能遗漏某些调用点，导致部分路径使用默认 model 而部分使用 override model。

缓解：Slice C 实现前先 grep 所有 `loop._model` / `self._model` 引用，列出完整改动清单。per-run model 作为 `RequestState` 的一个字段传播，而不是靠调用者逐个传参。
