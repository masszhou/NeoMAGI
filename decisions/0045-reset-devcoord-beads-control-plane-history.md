# 0045-reset-devcoord-beads-control-plane-history

- Status: accepted
- Date: 2026-03-02

## 选了什么
- 将当前 NeoMAGI 的 beads/devcoord 协作控制面视为试验期资产，正式退役其既有本地状态与远端历史，不做迁移、不做兼容保留、不继续修补旧控制面。
- 从一个全新的 beads 基线重新初始化未来协作控制面；新的 beads 仅承载后续项目治理与执行态，不再承担既往里程碑的长期历史账本职责。
- 重置后使用 beads 官方默认初始化与默认目录/数据库策略，不再保留 NeoMAGI 私有 beads 配置或私有目录约定。
- 将历史里程碑、实施教训、治理结论与验收证据的长期依据固定为 Git 提交与 Markdown 文档，包括：`decisions/`、`dev_docs/plans/`、`dev_docs/reviews/<phase>/`、`dev_docs/logs/<phase>/`、`design_docs/`。
- 将本次调整明确限定为项目治理层重置，不涉及产品代码、功能逻辑、数据库 schema、运行时行为或既有工作分支内容。
- 新控制面遵循“单项目、单 beads 控制面、轻装运行”的原则；实际数据库物理布局以 beads 当前发布版默认行为为准，项目侧不另行发明目录语义或数据库命名约定。

## 为什么
- beads 已经成功验证了对 NeoMAGI 真正关键的能力：在 `tmux` 中断、agent 崩溃或上下文丢失后，帮助 Agent Teams 找回状态、恢复推进并最终完成任务。
- 既有 beads 历史主要来自试验期探索，混入了 early-stage 拓扑试错、remote 试验、目录模型摇摆与操作面噪音；继续保留只会增加解释成本和后续运维复杂度。
- 项目长期需要对抗熵增。对抗熵增的正确方向不是无限背负试验期状态，而是把真正有价值的沉淀保存在更稳定、更可审阅的 Git 与 Markdown 文档中，把易腐蚀的执行态控制面保持尽可能轻。
- NeoMAGI 的核心进展并未依赖 beads 历史才能成立：代码已经在 Git 中，计划、审阅、总结与治理判断已经在 Markdown 中。放弃旧 beads 历史不会抹去项目成果，只是放弃试验期的执行态噪音。
- 对当前 beads 目录/remote 语义继续做项目侧补丁，会把上游试验期问题永久转化为本项目负担；与其修补历史，不如干净重启控制面，并把真正的长期经验沉淀进决议与文档。
- 采用 beads 默认初始化与默认目录/数据库策略，可以让 NeoMAGI 后续持续借力 beads 的产品演进与默认使用路径，避免再次形成项目私有分叉。

## 放弃了什么
- 方案 A：保留当前 beads 历史，继续修复 push/pull、remote 和目录错位问题。
  - 放弃原因：会把试验期噪音固化为长期资产，增加维护成本，不符合“轻装继续前进”的治理目标。
- 方案 B：迁移/清洗既有 beads 数据后再继续使用。
  - 放弃原因：迁移与筛洗本身就是额外复杂度，而且历史价值已经被 Git 和 Markdown 文档充分覆盖，没有必要再维护第二份清洗后的账本。
- 方案 C：在 NeoMAGI 本地增加 wrapper、patch 或特殊约定，绕过 beads 当前实现细节。
  - 放弃原因：这是将上游试验性问题下沉为项目私有债务，不利于后续升级与长期收敛。

## 影响
- 现有 `.beads` 控制面可被整体清空并重新初始化；旧本地状态和旧 beads remote 不再被视为项目正式历史基线。
- 后续若需要追溯过去阶段，以 Markdown 文档和 Git 记录为准，而不是以 beads 中的旧 issue/event 为准。
- 新 beads 控制面启动后，应只录入仍然与未来协作有关的最小治理信息，不回填旧里程碑，不补录试验期事件。
- 本决议本身应作为重置依据写入长期文档；后续任何“为何 beads 历史从某时点开始重新计算”的问题，都以本决议作为解释基线。
- 重置后如 beads 默认行为发生演进，NeoMAGI 原则上跟随上游默认，不再为兼容旧本地设定而保留项目私有特例。
